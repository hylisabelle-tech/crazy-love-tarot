# 기획 문서 템플릿

> 각 Phase에서 문서 작성 시 이 템플릿의 구조를 따른다.

---

## 1. project-summary 템플릿

```markdown
# {프로젝트명}

> {한줄 요약}

## 배경 및 목적

- 왜 만드는가 (문제/기회)
- 기대 효과

## 핵심 기능

우선순위 순으로 나열한다.

| # | 기능 | 설명 | 우선순위 |
|---|------|------|---------|
| 1 | ... | ... | 필수 |
| 2 | ... | ... | 필수 |
| 3 | ... | ... | 선택 |

## 대상 사용자

- 주요 사용자: ...
- 보조 사용자: ... (있다면)

## 기술적 범위

- 포함: ...
- 제외: ...
- 제약사항: ...

## 성공 기준 (선택)

- ...
```

### 작성 규칙

- 기능은 개조식으로, 한 항목에 하나의 기능만
- 우선순위는 `필수` / `선택` 2단계로 단순하게
- 기술적 범위에서 "제외" 항목을 명확히 하여 스코프 크립 방지

## 2. ux-flow 템플릿

```markdown
# {프로젝트명} — UX Flow

## 유저 시나리오

### 시나리오 1: {시나리오명}

- **사용자**: {누가}
- **목표**: {무엇을 달성하려 하는가}
- **플로우**:
  1. ...
  2. ...
  3. ...
- **성공 조건**: ...
- **예외 상황**: ...

### 시나리오 2: ...

## UX 플로우

Mermaid flowchart로 핵심 플로우를 시각화한다.

\```mermaid
flowchart TD
    A[진입점] --> B{분기 조건}
    B -->|조건1| C[화면1]
    B -->|조건2| D[화면2]
    C --> E[완료]
    D --> E
\```

## 정보 구조 (IA)

페이지/섹션의 계층 구조를 트리로 표현한다.

\```
{프로젝트명}
├── 메인
│   ├── 섹션 A
│   └── 섹션 B
├── 상세
│   ├── ...
│   └── ...
└── 설정
\```

## 데이터 모델

핵심 엔티티와 관계를 정의한다.

| 엔티티 | 주요 필드 | 관계 |
|--------|----------|------|
| ... | ... | ... |

## 컴포넌트 리스트

기존 디자인 시스템 컴포넌트 재활용을 우선한다.

| 컴포넌트 | 용도 | 구분 | 기존 경로 / 비고 |
|----------|------|------|-----------------|
| ... | ... | 재활용 | `components/card/...` |
| ... | ... | 신규 | 카테고리: layout |
| ... | ... | 수정 | 기존 + props 추가 필요 |
```

### 작성 규칙

- 시나리오는 핵심 플로우만 (3~5개 이내)
- Mermaid 다이어그램은 주요 분기만 포함, 세부는 시나리오 텍스트로
- 컴포넌트 리스트의 "구분" 컬럼: `재활용` / `수정` / `신규` 3단계
  - `재활용`: 기존 컴포넌트 그대로 사용
  - `수정`: 기존 컴포넌트에 props 추가/변경 필요
  - `신규`: 새로 만들어야 함 → 배치할 카테고리 명시
- 데이터 모델은 프론트엔드 관점의 상태/엔티티 중심

## 3. visual-direction 템플릿

```markdown
# {프로젝트명} — Visual Direction

## 톤앤매너

키워드 3~5개로 전체 방향을 정의한다.

- **키워드**: ... / ... / ... / ...
- **설명**: ...

## 컬러 방향

| 용도 | 현재 토큰 | 현재값 | 변경 방향 | 근거 |
|------|----------|--------|----------|------|
| Primary | `primary.main` | #... | ... | ... |
| Secondary | `secondary.main` | #... | ... | ... |
| Background | `background.default` | #... | 유지 | — |

## 타이포그래피 방향

| 요소 | 현재 설정 | 변경 방향 | 근거 |
|------|----------|----------|------|
| h1 | ... | ... | ... |
| body1 | ... | ... | ... |

## 간격 및 레이아웃

- spacing 기본 단위: ...
- 주요 레이아웃 패턴: ...
- 반응형 브레이크포인트 전략: ...

## 레퍼런스

사용자가 제공한 레퍼런스 이미지/사이트를 정리한다.

| # | 레퍼런스 | 참고 포인트 |
|---|---------|------------|
| 1 | {이미지 경로 or 설명} | {무엇을 참고할 것인가} |
| 2 | ... | ... |

## 변경 필요 토큰 요약

최종적으로 theme 파일에서 수정이 필요한 토큰 목록이다.

| 토큰 경로 | 현재값 | 변경값 | 적용 대상 |
|-----------|--------|--------|----------|
| `palette.primary.main` | ... | ... | ... |
| ... | ... | ... | ... |
```

### 작성 규칙

- 현재 테마 토큰값은 `mui-theme.md`에서 확인하여 정확히 기재
- "유지"하는 토큰도 표에 포함하여 의도적 유지임을 명시
- 레퍼런스 이미지는 사용자가 제공 — Claude가 임의 URL 생성 금지
- 변경 토큰 요약은 구현 시 theme 파일 수정의 직접적 입력이 됨
